# PicPeek Dry-Run Pilot Report

## Goal

Validate SignalLake SDK value with a real product-style PicPeek flow before expanding more infrastructure.

## Mode

- Dry-run SDK mode.
- No network by default.
- Local relay is called only by the pilot harness.
- Relay upload payload is encrypted with `signallake.encrypted-event-batch.v1`.
- No PicPeek repository files are modified in this phase.

## Event Coverage

| Event | Trigger | Privacy Notes |
| --- | --- | --- |
| `app.opened` | Tauri app process starts and primary window is ready. | No file, folder, image, or clipboard fields. |
| `screen.viewed` | Library / preview / settings route becomes visible. | Stable semantic screen id only. |
| `command.invoked` | Toolbar/menu/shortcut command. | Stable command id and entry only. |
| `error.occurred` | Handled app or SDK error path. | Stable error code only; no raw content. |

## Evidence Command

```bash
npm run pilot:picpeek
```

Acceptance:

- emits exactly four mapped events
- rejects forbidden folder path and clipboard text fields
- builds one encrypted upload batch
- relay upload returns `202`
- relay pull returns one delivery
- relay ACK returns `200`

## Current Result

The result is generated by `pilots/picpeek/run-dry-run.mjs` and covered by `tests/picpeek-pilot.test.mjs`.
The relay sees encrypted batch envelope metadata only. Event names and properties remain inside the encrypted payload.

## Next Product Decision

If this report is accepted, the next step is to enter the PicPeek repository and wire the same four semantic calls into the real Tauri app lifecycle and commands.
